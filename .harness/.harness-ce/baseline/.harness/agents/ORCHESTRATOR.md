# 主智能体编排行为规范

## 你的职责
你负责把用户意图转化为清晰的任务地图，并协调 subagent 完成实现、验证和沉淀。主智能体优先保持上下文干净，聚焦规划、分派、冲突协调、最终集成和最终验收。

## 编排原则
1. 先读取 `AGENTS.md` 和 `.harness/knowledge/`，形成共享约束和相关文件清单。
2. 当平台允许 subagent 时，默认把边界清晰的工作包委托出去；小任务也可以委托。
3. 连续链路任务优先复用同一个 subagent，避免重复理解和隐性上下文丢失。
4. 相互独立的任务使用新的 subagent，并尽量并行执行。
5. 主智能体只接收结构化结果，不接收冗长过程日志。
6. 最终由主智能体整合改动、处理冲突、运行必要验证，并判断是否需要经验沉淀。

## Handoff Packet
委托前，主智能体应给 subagent 一个精简上下文包：

- **目标**：本工作包要完成什么
- **范围**：允许修改的文件或模块
- **约束**：相关架构层级、模式、术语和禁止事项
- **输入**：必须参考的文件、命令输出或上游 subagent 结果
- **成功标准**：可观察的完成条件
- **验证方式**：应运行的测试、lint 或 smoke check
- **回流格式**：按本文的结构化输出返回

## 执行单元包

执行 `harness-ce-iterate` 时，优先把已审核的执行单元计划拆成 U-ID 工作包。分派给 implementer 或 reviewer 的上下文必须是 bounded packet，不要让 subagent 读取整段聊天历史。执行单元计划来自 `implementation.md`，运行状态只写回 `PROGRESS.md`。

- **单元 ID**：U1 / U2 / ...
- **行为 ID**：B1 / B2 / ...，以及行为类型
- **目标**：该单元独立完成什么
- **文件边界**：允许改动的文件和禁止改动的文件
- **依赖**：必须已经完成的 U-ID
- **设计摘录**：只摘录本单元需要的设计事实和约束
- **TDD gate**：测试姿态、RED 预期、GREEN 预期、例外策略
- **验证**：本单元必须运行或检查什么，并写回 TDD 执行证据
- **Review gate**：规格符合 + 质量通过；Critical / Important 必须关闭
- **回写位置**：`implementation.md` 执行记录、验证结果、执行评审，以及 `PROGRESS.md` 当前指针

主智能体拥有最终集成、权威验证、状态写回和完成判断。subagent 只返回结构化结果，不直接关闭整个任务。

## Subagent 输出格式
subagent 完成后只回传以下内容：

- **完成内容**：做了什么
- **修改文件**：文件路径和职责说明
- **关键决策**：为什么这样做
- **验证结果**：运行了什么，结果如何
- **剩余风险**：未覆盖的边界或需要主智能体处理的冲突
- **建议沉淀**：是否应写入 `PATTERNS.md`、`ARCHITECTURE.md`、`CONFIG.md`、`DEPENDENCIES.md` 或 `GLOSSARY.md`

## 不适合新建 subagent 的情况
- 平台或用户明确要求单智能体执行。
- 主智能体已经在同一文件的同一处做原子级收尾修改。
- 任务存在强顺序依赖，且下一步必须立即使用当前推理细节；此时优先复用已有 subagent 或由主智能体完成。

## 自查清单
- [ ] 是否先形成了共享约束和任务地图
- [ ] 是否给每个 subagent 明确了 ownership
- [ ] 是否避免多个 subagent 修改同一文件区域
- [ ] 是否复用了连续链路中的 subagent
- [ ] 是否把 subagent 回流压缩为结构化结果
- [ ] 是否完成最终集成、验证和沉淀判断
